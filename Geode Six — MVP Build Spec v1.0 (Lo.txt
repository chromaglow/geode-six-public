Acceptance Criteria — Read This First. This Is What "Done" Means.
Milestone 1 — Router

 POST /query accepts {prompt, user, sensitive, image_path} and returns a response from the correct model
 Image attachment routes to LLaVA
 Keywords biomass, pubmed, biotech, conversion route to BioMistral
 sensitive=true flag routes to Dolphin-Mistral
 Everything else routes to Llama 3.1
 Every request logged: timestamp, model used, latency ms, tokens in/out
 GET /health returns model availability and current memory usage within 2 seconds
 If available system RAM is below 1GB, /query returns: "System is busy, please try again in a moment" instead of timing out
 Router runs as systemd service, restarts on failure, starts on boot

Milestone 2 — Remote Access

 Charlie can reach the web UI from a different network via Cloudflare Tunnel URL
 PN can reach the web UI from a different network
 Neither Charlie nor PN can access SSH or any admin endpoint
 Email one-time PIN login works on mobile browser
 [ADMIN_USER] has full access to all endpoints
 cloudflared runs as a systemd service with Restart=on-failure

Milestone 3 — File Intake + Backup

 POST /gca/upload accepts PDF, DOCX, TXT, MD, XLSX, JPG, PNG
 AI assigns correct PROJECT code from: GEO, CMP, PR1, PR2, PR3, SYS, HR
 AI assigns correct TYPE code from: AGD, SUM, OPS, BRF, TSR, VIS, REQ, IDX, FRM, RPT, BIO, AGR, CON
 AI generates 3-4 word CamelCase description
 Date resolution follows this priority order:

1st: user provides date in note field
2nd: document metadata (PDF properties, DOCX created date, EXIF for images)
3rd: date found in original filename (parse formats: YYYYMMDD, MM-DD-YY, MonDD, etc.)
4th: file system creation date (ctime)
5th: today's date — flag in response as "Date estimated — please verify"


 File saved as [PROJECT]_[TYPE]_[Description]_[YYYYMMDD]_v[X.Y].ext — no spaces, no exceptions
 New uploads default to v0.1 unless user marks "ready to share" → v1.0
 Before committing file: check if a file with same PROJECT + TYPE + Description already exists in that folder. If yes, warn user: "A [version] of this file already exists. This will be saved as [next version]. Continue?" User must confirm before file is written.
 Upload response is a confirmation screen (not immediate commit) showing: assigned filename, project code, type code, description, date used, version, estimated flag if date was guessed. User can edit any field before confirming.
 On confirmation screen, all assigned fields are editable: project, type, description, date, version, ready-to-share toggle
 After user confirms, file is written to /mnt/nvme/gca/[PROJECT]/
 After file is written, Chroma embedding triggered automatically
 Unsupported file type returns plain-English error: "This file type is not supported. Accepted types: PDF, DOCX, TXT, MD, XLSX, JPG, PNG"
 rsync copies /mnt/nvme/gca/ to /mnt/usb-backup/gca/ every 15 minutes via cron
 USB not present → logs warning, does not crash, retries next cycle
 USB automounts on plug-in via udev rule for drive labeled GEODE-BACKUP

Milestone 4 — Semantic Search + Index Validation

 All files in /mnt/nvme/gca/ are chunked, embedded, and indexed in Chroma on first run
 New files are indexed automatically on upload confirmation
 POST /gca/search accepts {query, synthesize} and returns ranked results
 Each result includes: filename, path, project code, type code, relevance score, 200-char excerpt
 Natural language query returns relevant results (test: "Project Beta biomass funding" returns PR2 docs)
 synthesize=true passes top results to Llama 3.1 and returns a plain-English summary above the results list
 Chroma data persists at /mnt/nvme/chroma/ across reboots
 Weekly cron job runs scripts/validate_index.py — checks every Chroma entry against actual files on disk, removes entries where the file no longer exists, logs a report to /mnt/nvme/logs/index_validation.log

Milestone 5 — Web UI

 Single-page app with four sections: Assistant, Upload, Browse, Search
 Works on mobile Safari and Chrome (test at 390px width)
 Dark mode, clean minimal style, large tap targets

Assistant section:

 Text input, send button, model selector dropdown (Llama / BioMistral / Dolphin)
 Response displays below input, streams if possible

Upload section:

 Drag-and-drop or tap-to-upload
 Optional note field (user can hint project, topic, or date)
 Version toggle: Draft (v0.1) / Ready to Share (v1.0)
 After AI processes file: show confirmation screen with all assigned fields editable before committing
 If date was estimated: show yellow warning "Date estimated — please verify" next to date field
 If duplicate version detected: show warning banner before user confirms
 After confirmation: show success message with final filename

Browse section:

 Lists all files in GCA, sorted by date descending (newest first) by default
 Filter by project code (dropdown: All, GEO, CMP, PR1, PR2, PR3, SYS, HR)
 Filter by type code (dropdown: All, AGD, SUM, OPS, etc.)
 Sort options: date descending, date ascending, project A-Z
 Each row shows: filename, project badge, type badge, date, version
 Page loads under 3 seconds for up to 500 files

Search section:

 Natural language text input
 Results list: filename, project badge, type badge, relevance score, 200-char excerpt
 "Summarize results" toggle — calls synthesize=true, displays summary above results list
 Empty state: "No results found. Try different search terms."
 Full page loads under 5 seconds on LTE

GCA File Naming Convention — Implement This Exactly
Formula: [PROJECT]_[TYPE]_[Description]_[YYYYMMDD]_v[X.Y].ext
Example: PR1_TSR_EconomicDev_20260313_v2.0.docx
Project codes:

GEO — Geode Solutions internal operations
CMP — [YOUR_COMPANY_NAME] company-wide
PR1 — Project Alpha project
PR2 — Project Beta project
PR3 — Project Gamma project
SYS — Operations and Systems
HR — HR / People

Type codes:

AGD — Agenda
SUM — Summary / Meeting Notes
OPS — Operations
BRF — Brief
TSR — Teaser / Pitch material
VIS — Vision / Strategy
REQ — Requirements / Specs
IDX — Index / Master list
FRM — Framework / Model
RPT — Report / Research
BIO — Bio / CV
AGR — Agreement / Contract
CON — Contacts

Naming rules — enforce all:

Underscores between the 5 pieces only. No spaces anywhere in filename
Description is 3-4 words max, CamelCase (e.g., ProjectBetaBrief not project beta brief)
Date is always YYYYMMDD
Drafts start at v0.1. Files marked ready to share start at v1.0
Never overwrite — if file with that name exists, increment version
Keep original file extension

AI naming prompt — send this to Llama 3.1 on every upload:
You are a file naming assistant for [YOUR_COMPANY_NAME]. Given a document, assign:

PROJECT code (one of: GEO, CMP, PR1, PR2, PR3, SYS, HR)
TYPE code (one of: AGD, SUM, OPS, BRF, TSR, VIS, REQ, IDX, FRM, RPT, BIO, AGR, CON)
Description: 3-4 words, CamelCase, no spaces, describes what this specific document is about

If the user provides a hint note, use it to inform your decision.
Respond ONLY with valid JSON: {"project": "XXX", "type": "XXX", "description": "XxxXxx"}
No explanation. No markdown. JSON only.

0 2 * * 0 /usr/bin/python3 /path/to/geode-six/scripts/validate_index.py >> /mnt/nvme/logs/index_validation.log 2>&1
udev rule: create /etc/udev/rules.d/99-usb-backup.rules to automount any USB drive labeled GEODE-BACKUP to /mnt/usb-backup/. Include setup_usb_backup.sh that formats and labels the drive correctly.

After all files saved: runs full Chroma embedding pass

Index Validation Script

Rules For The Coding Agent

Build milestone by milestone. Do not start the next milestone until the current one passes all its acceptance criteria.
Run the relevant tests at the end of each milestone before moving on.
No build steps. UI is vanilla HTML/CSS/JS. No npm, no webpack, no React.
No Docker. Everything runs directly on the Jetson via systemd.
No spaces in any filename produced by this system. Ever. Hard rule.
All paths come from .env. Never hardcode /mnt/nvme/ anywhere in application code.
If nomic-embed-text is unavailable via Ollama, fall back to sentence-transformers with all-MiniLM-L6-v2 and note it in CHANGELOG.
The upload flow is two steps: AI suggests → user reviews and edits → user confirms → file is written. Never write a file before user confirmation.
Stop and ask only if you hit a genuine blocker: missing hardware info, ambiguous naming rule, or a dependency that will not install on ARM64. Otherwise proceed with stated decisions.
Log everything: every API request, every file intake, every backup run, every index validation. Logs go to /mnt/nvme/logs/.
Update CHANGELOG.md at the end of each milestone with what was built and what was tested.
